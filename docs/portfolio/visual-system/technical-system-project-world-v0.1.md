# Technical-system project world v0.1

## Purpose and hierarchy

Technical-system pages explain purpose, architecture, compatibility, platform, version, maintenance, implementation evidence, limitations, releases, and source availability. Limitations precede evidence in the production order so architectural constraints cannot be postponed to the end.

The technical shell and layout use a technical reading measure, compact metadata, inspectable evidence, and an architecture-first stage. Code remains short, relevant, semantic, and locally scrollable.

## Media, proof, and fallbacks

Real diagrams must be captioned and have meaningful alternatives. Metric evidence retains method, result, unit, and artifact together. Compatibility and versions remain textual and exact. No fake telemetry, terminal cosplay, dashboard chrome, glow, or fabricated benchmark appears.

Narrow screens and zoom keep one readable flow; code, tables, and diagrams contain their own overflow. Forced colors, print, reduced motion, and no JavaScript preserve all meaning. Authored marks are Level 0. The technical fixture, diagram/video transcript contract, long content, evidence, limitations, and release states are approval checks.
